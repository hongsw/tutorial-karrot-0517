## 1. 한 줄 요약

```
계약(인터페이스)을 먼저 정하고
 → 4명이 동시에 모듈을 짜고
  → 1명이 합쳐서 실제로 돌려보며 검증
```

서브에이전트 **5명**이 협업해 약 **4분 22초**(261초) 만에
`mail_source / classifier / responder / assigner / pipeline / run.py / README`까지 완성했습니다.

---

## 2. 실행 구조 (다이내믹 워크플로우)

"다이내믹 워크플로우"란, **어떤 에이전트를 몇 명 띄울지·무엇을 병렬로 돌릴지**를
코드(스크립트)로 직접 지휘하는 방식입니다. 사람이 짠 흐름대로 결정적으로 돌아갑니다.

```
┌──────────────────────────────────────────────┐
│  사전 작업 (오케스트레이터 = 메인 Claude)        │
│  · config.py / samples/inbox.json 준비          │
│  · "공통 인터페이스 계약(CONTRACT)" 작성          │   ← 모두가 같은 규격으로 짜게 만드는 핵심
└──────────────────────────────────────────────┘
                     │
        ┌────────────┴─── Phase 1: Implement (병렬) ───────────┐
        │            │            │            │
   ┌────▼───┐  ┌─────▼────┐  ┌────▼─────┐  ┌───▼──────┐
   │ Agent A │  │ Agent B  │  │ Agent C  │  │ Agent D  │   ← 4명이 "동시에" 작업
   │mail_src │  │classifier│  │responder │  │ assigner │
   └────┬───┘  └─────┬────┘  └────┬─────┘  └───┬──────┘
        └────────────┴──── 배리어(barrier) ─────┴──────┐
                     │  (4개 파일이 모두 생겨야 통합 가능)  │
        ┌────────────┴─── Phase 2: Integrate (단독) ──────────┐
        │   ┌──────────────────────────────────────────┐     │
        │   │ Agent E: pipeline.py + run.py + README.md │     │
        │   │  → 직접 `python run.py` 실행하며 디버그·검증 │     │
        │   └──────────────────────────────────────────┘     │
        └──────────────────────────────────────────────────┘
                     │
                 최종 산출물 + 실행 리포트
```

- **Phase 1 (병렬)**: 서로 **다른 파일**을 만들기 때문에 충돌이 없습니다 → 동시에 실행해도 안전.
- **배리어**: 통합은 4개 모듈이 **모두** 있어야 가능하므로, 여기서만 "다 끝날 때까지 대기"합니다.
- **Phase 2 (단독)**: 부품을 조립하고 **실제로 돌려보며** 고치는 일은 한 명이 책임지는 게 깔끔합니다.

## 3. 왜 "계약 먼저"인가 (병렬의 함정과 해법)

병렬로 짜면 빠르지만, 각자 멋대로 만들면 **부품이 안 맞습니다.**
(예: A는 `fetch_emails()`가 리스트를 주는데, B는 딕셔너리를 기대 → 조립 실패)

그래서 오케스트레이터가 **먼저 규격서를 못 박았습니다:**

```text
# 이메일 레코드
{"subject": str, "sender": str, "date": str, "body": str}

# 분류 결과
{"category": str, "priority": "높음|보통|낮음", "summary": str, "method": "rule|llm"}

# 함수 시그니처 (4개 모듈 모두 이 규격대로)
fetch_emails(source, limit, unread_only, mailbox) -> list[dict]
classify(email) -> dict
generate_reply(email, classification, assignee) -> str
class Assigner: assign(category) -> str
```

이 계약을 4명 모두에게 같이 전달 → **결과적으로 통합 단계에서 코드 수정 0건**.
(부품이 처음부터 정확히 맞아떨어졌습니다.)

## 4. 효과 — 순차 vs 병렬

| 구분 | 순차 처리(1명이 다) | 병렬 워크플로우(이번 방식) |
|---|---|---|
| 모듈 작성 | 7개 파일을 한 줄씩 차례로 | 4개 핵심 모듈 **동시** 작성 |
| 벽시계 시간 | 모듈 시간의 **합** | 가장 느린 1개 + 통합 (≈ **4분**) |
| 일관성 | 자연스럽게 맞음 | **계약**으로 강제 (이번엔 수정 0건) |
| 검증 | 마지막에 한 번 | 통합 에이전트가 **직접 실행**해 디버그 |

> 핵심: 병렬의 벽시계 시간은 **"작업들의 합"이 아니라 "가장 오래 걸린 하나"**에 가깝습니다.
> 독립적인 일이 많을수록 이득이 커집니다.

### 이번 실행 실측치
- 투입 에이전트: **5명** (병렬 4 + 통합 1)
- 총 토큰: 약 **363,854** tokens
- 도구 호출: **32회**
- 소요 시간: **261초 (≈4분 22초)**
- 통합 중 버그 수정: **0건** (계약 일치 덕분)

## 5. 언제 이 방식이 좋은가 / 나쁜가

**좋다 (병렬 추천)**
- 작업이 **독립적인 여러 조각**으로 쪼개질 때 (모듈/파일별)
- 각 조각이 충분히 무거워서 동시에 돌릴 가치가 있을 때
- 인터페이스를 미리 명확히 정의할 수 있을 때

**나쁘다 (그냥 순차로)**
- 조각들이 **서로 강하게 얽혀** 한 명이 보면서 짜야 할 때
- 같은 파일을 여러 명이 동시에 고쳐야 할 때 (충돌 위험 → 굳이 한다면 격리 필요)
- 일이 작아서 병렬 준비 비용이 더 클 때
